## 1. 安全审计驱动的架构重构

### 教训: 审查代码 ≠ 改代码，但要改就改根

审计发现 13 个问题后，最初只想打补丁。但 `sharedSecret=""` 导致整个同步协议形同虚设——不修根源，修什么都没用。

**正确做法**: 把配对协议从头重写（短码验证 + 指纹密钥交换），同时串联已有的 `AcpCrypto` 和 `PromptFirewall`——这两者一直存在但从没被用过。

### 教训: 安全模型选否认清单，不要选允许清单

最初给 `agent.read` 做了白名单（只允许 3 个目录），Agent 连 `/data/local/tmp/` 都读不了。用户指出"应该是禁止Agent编辑APK核心文件，而不是把其他目录都禁止"。

**否认清单**:
- `agent.read` → 无限制
- `agent.write` → 只禁 `/system/` `/vendor/` `/data/app/` `/data/dalvik-cache/`

这才是 Agent 需要的自由度。

## 2. Android 14 权限陷阱

### 教训: 不要假定权限存在

华为 MagicPad 13 (Android 14) 上根本没有 `READ_EXTERNAL_STORAGE` 权限——它被 `maxSdkVersion=32` 从 Manifest 里抹掉了。用户在设置里找不到这个权限，因为**系统就不提供**。

**正确做法**: 用 `/data/local/tmp/` 做桥接通道。——不需要权限，ADB 和 App 都能读写。

### 教训: Manifest 的 `maxSdkVersion` 是双刃剑

```
READ_EXTERNAL_STORAGE  maxSdkVersion=32   ← Android 13+ 不申请
WRITE_EXTERNAL_STORAGE maxSdkVersion=29   ← Android 11+ 不申请
```

看似合理（新版本不需要旧权限），但导致 `/sdcard/Download/` 在华为平板和 vivo 手机上都无法访问。**要么去掉限制，要么提供无权限的备用通道。**

## 3. 系统提示词是 Agent 的"圣经"

### 教训: 改了 CLI 命令，必须同步改系统提示词

加了 `agent.read`/`agent.write` 后，Agent 仍然尝试用 `fs.cat`/`fs.write`。因为系统提示词里到处都是硬编码的 `fs.*` 命令：

- `PromptEngine.kt` — few-shot 示例用 `fs.ls`
- `source-index.md` — "仍不足时，用 `fs.cat` 或 `fs.ls`"
- `filesystem.md` — 整篇文档都是 `fs.*`
- `SkillPlugin.kt` — 7 处代码创建指导用 `fs.write`
- `tavily.md`、`make-skill.md` — 输出指令用 `fs.write`

修复需要改 **8 个文件**，不是只加一个新命令就够了。

### 教训: LazyColumn key 必须全局唯一

`ChatMessageUi.User.stableId = "u_${content.hashCode()}"`——相同内容产生重复 key→Compose 崩溃。`System.nanoTime()` 解决。

## 4. 协议统一的价值

### 教训: ACP 已有 18 种消息类型，但只有 1 个 Handler

`DELEGATE`、`SHARE_MEMORY`、`SHARE_SKILL` 收到了直接返回空 ack——从未被处理。加上 4 个新 Handler 后，ACP 从"只能做孪生同步"变成了通用 Agent 通信协议。

### 教训: MCP 和 ACP 应该是一体的

两套协议共存但零集成。`McpOverAcpBridge` 用 98 行代码把 MCP JSON-RPC 包装进 ACP，Claude Code 就能用标准 MCP 协议调用 Agent 的浏览器自动化工具。

### 教训: HTTP 响应别丢 data 字段

`AcpHttpTransport` 的响应只序列化了 `result.message` 和 `success`，把 `data` 丢了。sync query 返回了正确结果但 Claude Code 看不到——修了 3 行就让整个同步查询模式能用了。

## 5. 开发者体验与跨工具通信

### 教训: Agent 应该有内置文件 I/O

让 Agent 读文件需要先装 `fs` 插件——这就像让新电脑用户先装驱动程序才能打开文件夹。`agent.read`/`agent.write` 作为内置命令解决了这个问题。

### 教训: 两台真机调试的边际成本

每轮调试 = 改代码 → 构建(3-4分钟) → 装 2 台设备 → 激活 → 配对。设备还经常断连、需要屏幕确认。**用 `adb logcat -s TAG` 比肉眼找崩溃快 100 倍**（这是 dev guide 里的教训，这次再次验证）。

### 教训: ADB forward 比文件桥接可靠

文件桥接（`/sdcard/Download/mengpaw-task.md`）有两个致命问题：(1) Android 14 无存储权限读不了 (2) 需要用户通过文件提交功能手动送文件。`adb forward tcp:9876` + ACP HTTP 直接通信，延迟从轮询的 30 秒降到同步响应的 1 秒。
